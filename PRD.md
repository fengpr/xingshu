# 星枢 PRD

## 1. 文档信息

- 产品名称：星枢（Xingshu）
- 产品定位：混合式全功能 AI 私人助手
- 文档版本：v1.0
- 文档用途：作为后续架构设计、任务拆解、开发排期、测试验收的产品基线
- 参考来源：`D:\Download\星枢·混合式全功能AI私人助手设计方案（Python+Java·个人开发·生产级·多智能体进阶版）.docx`

## 2. 产品概述

星枢是一款面向个人用户的桌面端 AI 私人助手，采用“本地隐私优先 + 按需联网增强 + 多智能体协同”的混合架构，兼顾数据安全、功能丰富度与自动化能力。产品以个人开发可落地为前提，优先覆盖高频、可复用、可持续迭代的核心场景，包括知识管理、办公自动化、开发辅助、设备控制、多模态交互与插件扩展。

产品首期形态为跨平台桌面客户端，面向 Windows 为主、兼容 macOS / Linux。系统通过 Python 承担 AI 能力与交互逻辑，通过 Java 承担安全、调度与服务支撑，逐步演进为具备多智能体协同能力的生产级 AI 助手。

## 3. 目标与核心价值

### 3.1 产品目标

1. 为个人用户提供一个可长期使用的本地优先 AI 助手，解决信息管理、重复劳动、内容生产与电脑操作自动化问题。
2. 在不牺牲隐私安全的前提下，引入联网增强能力，提升实时信息获取、复杂推理与多模态处理效果。
3. 建立“主智能体 + 专项智能体”的协同框架，为复杂任务拆解、插件生态与后续商业化打下基础。
4. 保证个人开发可控，支持分阶段落地、逐模块交付与持续迭代。

### 3.2 核心价值

- 隐私可控：敏感数据本地处理，联网能力可开关、可审计、可限权。
- 效率提升：覆盖知识检索、内容生成、办公处理、设备控制等高频场景。
- 能力弹性：离线可用，联网更强，兼顾基础稳定性和能力上限。
- 持续扩展：支持插件化扩展，并可平滑升级为多智能体协同系统。
- 个人可落地：技术选型务实，优先成熟组件与轻量架构，降低实现复杂度。

## 4. 用户角色

### 4.1 核心用户

- 个人知识工作者：需要整理资料、生成文档、处理日常办公任务。
- 开发者：需要代码生成、纠错、文档辅助、本地工程分析与技术检索。
- 学习者 / 创作者：需要总结资料、翻译、写作润色、思维整理与多模态辅助。

### 4.2 扩展用户

- 高隐私敏感用户：对本地存储、断网能力、日志可控有强需求。
- 自动化重度用户：希望通过快捷键、悬浮窗、定时任务、设备控制提升效率。
- 插件开发者：希望基于标准接口扩展功能、发布插件或专项智能体。

### 4.3 用户角色诉求

- 普通用户：上手简单、响应稳定、结果可直接使用。
- 进阶用户：可配置模型模式、联网权限、插件能力、智能体规则。
- 开发者用户：接口清晰、扩展成本低、版本兼容性稳定。

## 5. 产品范围

### 5.1 本期范围

- 桌面端 AI 助手主应用
- 本地 / 联网双模型切换
- 私人知识库
- 生产效率工具
- 本地设备控制
- 多模态输入输出
- 在线插件中心
- 基础多智能体调度与管理

### 5.2 暂不纳入本期

- 复杂分布式部署
- 自研模型训练与微调平台
- 大规模开放式插件审核平台
- 重度移动端原生能力

## 6. 核心功能清单

### 6.1 模块一：私人知识库

#### 功能目标

让助手理解用户的本地资料、长期偏好与个人上下文，提供可追溯、可编辑、可扩展的问答与检索能力。

#### 核心功能

- 多格式导入：支持 PDF、Word、Excel、TXT、Markdown、图片 OCR、音频转文字。
- 文档管理：支持分类、标签、时间维度管理与批量导入。
- 本地检索：基于关键词与语义检索返回文档片段、摘要与来源定位。
- 知识编辑：支持在系统内追加笔记、标注重点、更新索引。
- 记忆能力：记录用户偏好、习惯、常用输出格式与历史上下文。
- 混合问答：结合本地知识与联网搜索结果生成综合答案。
- 文档分级加密：支持敏感资料的单独权限保护。

#### 结果要求

- 默认优先使用本地数据。
- 返回结果需展示引用来源或所属文档。
- 敏感内容未经授权不得进入联网链路。

### 6.2 模块二：生产效率引擎

#### 功能目标

通过 AI 自动化处理高频重复任务，覆盖办公、开发、学习与创作三类场景。

#### 核心功能

- 办公自动化：Excel / Word / PDF 批处理、格式转换、文件整理、表单填充、邮件辅助。
- 开发辅助：代码生成、错误修复、代码解释、注释生成、接口调试、SQL 辅助。
- 学习 / 创作辅助：周报、方案、邮件、总结、翻译、润色、思维导图、错题整理。
- 联网增强：补充 API 文档、博客、案例、素材、题库等在线内容。
- 场景推荐：结合用户行为推荐相关操作，如图表生成、待办提取、模板套用。

#### 结果要求

- 输出内容可直接复制、导出或二次编辑。
- 办公与开发任务需保留输入与结果关联，便于复盘。

### 6.3 模块三：本地设备控制

#### 功能目标

让助手具备“电脑管家”能力，帮助用户通过自然语言完成常见系统与文件操作。

#### 核心功能

- 应用控制：打开、关闭、切换常用软件。
- 系统控制：音量、亮度、分辨率、锁屏、关机、重启等操作。
- 文件操作：打开文件、创建目录、批量重命名、基础整理。
- 定时任务：提醒、定时启动应用、定时执行固定操作。
- 屏幕理解：截图、OCR 识别、基于界面的简单自动化。
- 场景动作：如“开始工作”“进入休息模式”等组合指令。

#### 结果要求

- 高风险操作需要二次确认。
- 设备控制默认离线执行，不依赖网络。

### 6.4 模块四：多模态交互

#### 功能目标

提供文字、语音、图片、文件的统一交互体验，降低使用门槛，提升复杂任务输入效率。

#### 核心功能

- 文本对话：主入口能力，承载大部分任务表达。
- 语音输入输出：离线语音识别、语音合成、连续对话。
- 图片理解：识别图片内容、OCR 提取、基础图片问答。
- 文件拖拽：拖入文档后直接发起问答、总结或处理。
- 联网高级能力：AI 绘画、视频解析、多语言语音、语音克隆等。
- 跨模态任务：如“语音指令 + Excel 文件”联合触发分析流程。

#### 结果要求

- 交互方式切换应统一在一个任务流中完成。
- 多模态结果需要支持导出与复制。

### 6.5 模块五：在线插件生态

#### 功能目标

通过在线插件市场扩展系统边界，为后续专项智能体生态提供分发基础。

#### 核心功能

- 插件市场：列表、分类、搜索、筛选、评分。
- 插件管理：安装、更新、卸载、启停、版本适配。
- 插件模板：提供 Python 插件开发模板与基础接口规范。
- 智能体扩展：部分插件可升级为专项智能体。
- 插件分发：支持云端仓库、下载、校验与本地动态加载。

#### 结果要求

- 插件安装不应破坏主程序稳定性。
- 插件需具备基础元数据、版本与权限声明。

### 6.6 模块六：多智能体协同与系统能力

#### 功能目标

建立从单助手到多智能体系统的基础调度能力，支持复杂任务拆解、协同执行与结果汇总。

#### 核心功能

- 主智能体调度：任务拆解、路由分发、结果整合。
- 专项智能体：文档、检索、办公、代码、设备控制、多模态、联网、同步等。
- 智能体管理：启停、优先级、协同规则配置。
- 模式切换：仅本地、优先联网、全程联网。
- 联网控制中心：联网开关、白名单、日志审计、记录清理。
- 数据安全：密钥加密、敏感词过滤、隐私分级隔离。
- 同步与备份：本地备份、非隐私云同步、配置恢复。
- 日志统计：使用日志、调用统计、异常监控、协同记录。

#### 结果要求

- 复杂任务需可追踪执行路径。
- 本地智能体与联网智能体必须遵守统一隐私规则。

## 7. 关键使用场景

1. 用户导入本地资料后，询问“基于我的项目总结，帮我整理一版周报”，系统优先检索本地知识库并生成可编辑内容。
2. 用户拖入 Excel 文件并输入“帮我做数据分析并生成图表”，系统调用数据处理与可视化能力输出结果。
3. 用户说“我要开始工作”，系统自动打开办公软件、整理窗口并进入专注模式。
4. 用户输入“结合我的本地笔记和今天最新资讯，总结 AI Agent 方向趋势”，系统执行混合检索并输出带来源的总结。
5. 用户输入“帮我分析这个仓库的接口结构并补充接口文档”，系统结合本地代码与模型能力完成开发辅助任务。

## 8. 产品规则与约束

### 8.1 隐私与安全规则

- 本地文档、聊天记录、设备控制指令、API 密钥默认本地存储。
- 联网请求需可关闭、可审计、可清理。
- 敏感文本在出网前需做过滤或脱敏。
- 高危系统操作必须二次确认。

### 8.2 体验规则

- 默认提供桌面主界面、悬浮窗、快捷键三种入口。
- 用户可明确感知当前模式：本地、混合、联网。
- 重点任务需要可中断、可查看状态、可复用历史结果。

### 8.3 交付原则

- 先落地高频刚需，再扩展惊艳能力。
- 优先复用成熟组件，避免早期过度自研。
- 功能设计需服务于单人开发的可实施性。

## 9. 技术栈约束

## 9.1 总体约束

- 必须采用 Python + Java 双栈架构，不引入第三核心后端语言。
- 必须支持本地优先运行，联网能力为增强项而非唯一依赖。
- 必须支持跨平台桌面端交付，首要兼容 Windows。
- 必须为后续多智能体协同预留统一调度与通信接口。

### 9.2 Python 侧约束

- 负责 AI 模型接入、桌面 UI、多模态处理、知识库业务、插件系统、专项智能体实现。
- 桌面端 UI 首选 PyQt6。
- 本地模型接入首选 Ollama。
- 文档处理、多模态、自动化尽量采用成熟 Python 库，不重复造轮子。
- Python 侧不承担核心加密与权限管控策略的最终兜底。

### 9.3 Java 侧约束

- 负责本地服务中间层、跨语言通信、安全管控、存储支撑、日志监控、主智能体调度。
- 需提供稳定的本地 Socket 或 HTTP 服务，作为 Python 与系统能力之间的桥梁。
- 需承担 AES-256 密钥加密、敏感词过滤、联网请求治理、权限与日志审计。
- 不承担复杂模型推理逻辑与 UI 主实现。

### 9.4 数据与存储约束

- 本地数据库首选 SQLite。
- 敏感数据必须加密存储。
- 云端仅同步非隐私配置、插件列表、模板、收藏与公共配置。
- 存储结构需支持智能体配置、协同规则与日志记录。

### 9.5 模型与联网约束

- 本地模型需支持离线运行，优先轻量模型。
- 在线模型通过标准 API 对接，至少支持可替换的多供应商接入。
- 需要支持自动与手动两类模型切换策略。
- 网络故障或用户断网时，系统应自动降级到本地可用能力。

### 9.6 插件与智能体约束

- 插件接口需标准化，支持安装、更新、卸载、动态加载。
- 插件需要权限描述与版本信息。
- 智能体能力需复用插件与模块能力，避免重复建设。
- 主智能体与专项智能体的通信协议需要保持统一、可扩展。

## 10. 非功能需求

### 10.1 性能

- 基础模式需支持秒级启动。
- 常见离线问答与文档检索应在可接受范围内返回结果。
- 闲置智能体与联网模块应支持按需加载与资源释放。

### 10.2 稳定性

- 异常任务不能拖垮主程序。
- 插件错误需隔离。
- 联网失败、模型不可用、解析失败等异常需提供清晰反馈。

### 10.3 可维护性

- 模块边界清晰，接口文档明确。
- 功能需支持逐模块迭代与替换。
- 保留日志、状态与错误上下文，便于调试与回归测试。

## 11. 分阶段建设建议

### 第一阶段：离线可用

- 打通 Python / Java 通信
- 完成主界面、基础问答、知识库导入检索、简单设备控制
- 落地 1 个主智能体与 2 个专项智能体雏形

### 第二阶段：混合增强

- 接入在线模型与联网搜索
- 增加基础多模态能力
- 完成联网控制中心与基础插件市场

### 第三阶段：协同完善

- 增强专项智能体协同
- 完善插件生态、个性化配置、多设备同步
- 补齐日志、升级、异常监控与系统打磨

## 12. 成功判定标准

- 用户可在本地模式下完成基础问答、资料检索、设备控制。
- 用户可在混合模式下获得本地知识与在线信息结合的增强结果。
- 核心高频场景可稳定运行：知识库、办公辅助、开发辅助、多模态输入。
- 智能体调度具备最小可用能力，能够处理简单复合任务。
- 产品具备持续扩展为插件生态和多智能体系统的结构基础。
