# 开发与文档同步技能

在星枢项目中开发任何功能时，始终遵循“代码与文档同步交付”的原则。

## 核心原则

- 以原子化提交为默认要求
- 完成任何核心功能开发后，立即更新对应文档，再结束当前任务
- 完成任何接口修改后，立即更新接口文档或接口注释，再结束当前任务
- 禁止只修改代码、不更新文档
- 如果变更影响用户使用方式，必须同步更新 `README.md`
- 如果变更影响工程规范或架构边界，必须同步更新 `SPEC.md`

## 原子化提交规则

- 将“代码修改 + 文档更新”视为一个不可拆分的交付单元
- 核心功能开发完成时，至少检查以下文档是否需要更新：
  - `README.md`
  - `PRD.md`
  - `SPEC.md`
  - `docs/API.md`
  - 对应模块说明文档
- 若本次提交包含接口字段、返回结构、调用方式、权限要求、错误码或路径变更，必须同步更新接口文档
- 若文档未更新，则视为功能未完成

## README 更新规则

### 新增功能

- 如果增加了新功能，必须在 `README.md` 的“功能特性”章节添加对应描述
- 描述应说明功能用途，不写实现细节，不堆砌内部术语

### 新增环境变量

- 如果增加了新的环境变量，必须在 `README.md` 的“环境配置”章节补充说明
- 说明至少包含：
  - 变量名
  - 用途
  - 是否必填
  - 默认值或示例值

### 启动或部署方式变更

- 如果改变了启动方式、构建方式或部署方式，必须同步更新 `README.md` 的“快速开始”章节
- 如果命令、前置依赖、运行顺序发生变化，必须更新示例命令

## 技术文档同步规则

### API 变动

- 如果是 API 变动，更新 `docs/API.md`
- 如果后续项目接入 Swagger 或 OpenAPI，必须同步更新对应注释或规范文件
- API 文档至少说明：
  - 接口路径
  - 请求方法
  - 请求参数
  - 返回结构
  - 错误场景

### 架构变动

- 如果是架构变动、模块边界变动、目录结构调整、通信协议调整、命名规范调整，必须更新 `SPEC.md`
- 若变动影响整体系统设计或模块职责，可同时补充 `docs/ARCHITECTURE.md`

### 产品范围或功能边界变动

- 如果新增、删减或重定义核心功能范围，评估是否需要更新 `PRD.md`
- 若变更影响产品目标、核心价值、角色范围或功能边界，必须更新 `PRD.md`

## 文档语言要求

- 所有面向用户的文档必须使用中文
- 所有操作手册、README、使用说明、注释说明优先使用中文
- 表述必须专业、简洁、易懂
- 避免使用含糊措辞，如“差不多”“可能”“后面再说”
- 避免写成口语化流水账，优先写明确结论和执行方式

## 开发执行流程

1. 先阅读相关需求文档与规范文档
2. 明确本次变更影响范围：功能、接口、配置、架构、文档
3. 先修改代码，再立即补齐对应文档
4. 自查 `README.md`、`SPEC.md`、`docs/API.md`、模块说明是否遗漏
5. 确认代码与文档一致后，再视为任务完成

## 提交前检查清单

- 是否新增了功能描述，但未更新 `README.md`
- 是否新增了环境变量，但未更新 `README.md`
- 是否修改了接口，但未更新 `docs/API.md`
- 是否修改了架构或目录，但未更新 `SPEC.md`
- 是否修改了产品边界，但未更新 `PRD.md`
- 是否新增了用户可见行为，但未补充中文说明

若以上任一项答案为“是”，先补文档，再结束任务。
