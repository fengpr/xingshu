# 前端界面美化技能

在实现任何前端页面或组件前，先明确视觉方向，再开始编码。不要先堆组件、后补样式。

## 设计思考

在编码前，先判断以下内容：

- 目的：这个界面要解决什么问题，服务谁
- 气质：选择一个明确风格，不要中庸摇摆
- 约束：技术栈、性能、可访问性、适配范围
- 记忆点：用户最终会记住什么

可选风格方向包括但不限于：

- 极简克制
- 编辑感 / 杂志感
- 未来科技
- 复古未来
- 工业工具感
- 奢华精致
- 柔和自然
- 玩具感 / 轻松活泼
- 粗野主义
- 几何装饰风

关键要求不是“复杂”，而是“有明确方向且执行一致”。

## 编码要求

输出必须满足以下要求：

- 代码可运行，不只给视觉描述
- 风格统一，有完整的视觉观点
- 细节打磨到位，包括排版、间距、层次、状态和反馈
- 优先产出可直接进入项目的生产级代码

## 审美约束

### 排版

- 优先选择有辨识度的字体组合
- 避免默认系统字体堆叠带来的普通感
- 标题字体与正文字体应有层次区分
- 用字号、字重、字距、留白建立信息秩序

### 色彩

- 先定义主色、强调色、背景层级和文本层级
- 优先用 CSS 变量统一颜色体系
- 颜色要有主次，不要平均分配视觉注意力
- 避免常见的白底紫渐变套板

### 动效

- 动效应服务于层次、节奏和反馈
- 优先做高价值动效，如首屏进入、卡片悬停、滚动揭示
- 页面内动效风格必须统一，不要零散堆砌
- React 场景下，若项目已具备相关依赖，可优先使用 Motion 类方案

### 空间与布局

- 允许使用非对称布局、交叠、偏移和打破常规网格的方式
- 极简风格依赖精确间距与克制留白
- 丰富风格依赖层次、节奏和视觉焦点

### 背景与氛围

- 不要默认使用单一纯色背景
- 可按风格加入渐变、纹理、噪点、图形、边框、投影、玻璃层或图案
- 所有装饰元素必须与整体风格一致，不要为了“丰富”而堆砌

## 明确避免

不要使用以下常见问题做法：

- 机械化、模板化的通用 AI 风格布局
- 过度依赖常见默认字体
- 没有主题意图的安全色板
- 所有页面都长得像同一套后台模板
- 动效很多但没有节奏与重点
- 只做表面换色，不调整结构、层次与空间

## 与项目配合方式

- 若任务发生在星枢现有界面中，优先保持与当前产品信息架构一致，再做美化
- 若是新页面，可大胆建立独立视觉方向，但需保留项目级命名与组件组织规范
- 实现完成后，如本次新增了用户可感知的新界面能力，同步更新 `README.md`
- 若因前端能力新增引入了新的目录约定或结构变更，同步更新 `SPEC.md`
